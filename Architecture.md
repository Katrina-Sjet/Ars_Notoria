# 프로그램 명세

본 프로그램은 <시계탑에 흐른 피> 보드게임을 진행함에 있어 “마도서” 기능 일체를 담당하고, 나아가 스크립트 저장, 캐릭터 능력 자동화, 히스토리 저장 등의 추가적인 기능을 구현한다.

**1. 시스템 개요 및 아키텍처 (System Overview)**

- **목적 및 운영 환경:** 본 프로그램은 <시계탑에 흐른 피>의 원활한 진행과 정보 가시성을 확보하기 위해 설계된 웹 애플리케이션입니다. 스토리텔러(ST)는 화면 비율이 다양한 태블릿이나 모바일 기기(혹은 노트북)를 통해 조작하며, 플레이어(PL)들은 일반적인 16:9 해상도의 대형 스크린(모니터 또는 TV)을 통해 정보를 열람합니다.
- **화면 분리 및 파일 구조:** 물리적인 스포일러 방지와 렌더링 최적화를 위해 애플리케이션은 두 개의 독립된 HTML 문서로 분리됩니다.
    - `Grimoire.html` (마도서): 게임의 모든 룰과 상태(엔진)를 관장하며 ST의 입력을 처리하는 중앙 통제 패널입니다.
    - `Viewer.html` (뷰어): 자체적인 로직 없이 전달받은 상태 데이터만을 시각적으로 렌더링하는 클라이언트입니다.
- **통신 아키텍처 (초경량 로컬 웹소켓):** 외부 인터넷 연결(WAN)이 완전히 단절된 환경에서도 구동 가능하도록, **근거리 통신망(LAN) 기반의 폐쇄형 통신 구조**를 채택합니다.
    - 대형 스크린이 연결된 메인 PC에서 Node.js 기반의 경량 중계 스크립트(`server.js`)를 백그라운드 환경으로 구동합니다.
    - ST의 마도서 기기와 PL의 뷰어 기기는 동일한 공유기(Wi-Fi) 내에서 할당된 내부 IP 주소를 통해 웹소켓(WebSocket)으로 접속하여 데이터를 실시간 동기화합니다.
- **정보 보안 및 송출 정책:** * **게임 진행 중:** 마도서(`Grimoire.html`)에서 뷰어(`Viewer.html`)로는 생존/사망 여부, 누적 투표수, 현재 위상(낮/밤) 등 철저히 검열된 '공개 정보'만을 브로드캐스트합니다. ST의 조작 실수로 인한 치명적인 역할 스포일러를 원천 차단합니다.
    - **게임 종료 시:** ST가 마도서에서 '게임 종료'를 선언하면, 그제서야 각 플레이어의 실제 역할과 게임 중 발생한 모든 야간 행동 히스토리 등 전체 상태 데이터를 뷰어로 일괄 전송하여 복기(Debriefing) 화면을 렌더링합니다.

**2. 데이터 관리 및 초기 세팅 (Data & Setup)**

**A. 데이터 출처 및 모델링 분리 원칙**

데이터는 그 성격과 시스템 개입 기여도에 따라 세 가지 출처로 완벽히 분리되어 관리됩니다.

1. **외부 JSON (코어 데이터 베이스):** 스크립트 메타데이터, 캐릭터 ID, 진영(Team), 캐릭터 유형, 그리고 **아이콘 URL**을 제공하는 정적 뼈대입니다.
2. **외부 CSV (텍스트/UI 오버라이드):** 구글 스프레드시트 URL을 통해 실시간 `fetch`하여 덮어씁니다. 번역 이름, 능력/밤 설명, **징크스(Jinx) 텍스트** 등 시스템 규칙에 관여하지 않는 순수 언어 및 시각 데이터만 전담합니다.
3. **내부 JS 클래스 (시스템 제어 및 로직):** 각 캐릭터별 `BaseLogic` 상속 파일 내부에서 코어 로직을 캡슐화합니다. **행동 로직, 리마인더 부착 규칙, 징크스(Jinx) 논리, 특수 셋업 변수, 위장(`disguisedAs`) 및 풀 제외(`excludeFromBag`) 속성** 등이 이곳에 하드코딩됩니다.

**B. 객체 속성 명세**

- **Character (캐릭터):** ID, 이름(CSV), 진영(JSON), 유형(JSON), 아이콘(JSON), 설명 텍스트(CSV), 셋업 변수(JS), 능력 로직(JS), 리마인더(JS), 징크스 로직(JS).
- **Player (토큰 인스턴스):** ID, 좌석 번호, 중앙 기준 좌표(x, y), 닉네임, 캐릭터 인스턴스, 생사 여부, 피지목 여부, 유령 투표권 상태, 보유 리마인더 배열.
- **파생 상태 (Derived State) 원칙:** 플레이어의 **'중독/취함/집착(Madness)/위장(Disguised)'** 등의 각종 상태는 독립된 변수(Boolean)로 관리하지 않습니다. 오직 현재 보유 중인 리마인더 배열을 스캔하여 동적으로 상태를 도출합니다.

**C. 특수 캐릭터 및 셋업 검증 규칙 (Setup Validation)**

- **스크립트 체급 자동 판정:** 로드된 정규 캐릭터(여행자/전설 제외)가 15개 미만이면 틴시빌(5~6인 전용), 15개 이상이면 풀시트(7~15인)로 묶어 유효 인원수를 강제합니다.
- **정규 인원수 동적 계산:** 기준 인원수는 마도서의 총 토큰 수에서 선택된 여행자(Traveler) 수를 차감하여 실시간 산출합니다.
- **다중 분기 셋업 변수 (Branching Modifiers):** `setupModifier`(선택 시 발동)와 `unselectedModifier`(미선택 시 발동, 예: 보초/둔세자)는 다수의 경우의 수를 담은 배열을 반환합니다. 검증기는 모든 가능성을 취합하여 유효성을 판별합니다.
- **명시적 위장 지정 (`disguisedAs`):** 주정뱅이, 미치광이 등이 스크립트 탭에서 선택되면 타겟 지정 모달이 즉시 호출됩니다. ST가 풀 내부의 '가짜 역할'을 명시적으로 지목하며, 지목된 가짜 역할의 원본 셋업 변수는 연산에서 완전히 무효화됩니다.
- **물리적 배분 제외 (`excludeFromBag`):** 릴몬스타 등 주머니에 들어가지 않는 캐릭터는 해당 속성을 부여하여 셋업 목표치만 비틀고 실제 물리적 카운팅에서는 0개로 배제됩니다.

**D. UI 레이아웃: 3-Tab 병렬 워크플로우**

- **마도서 탭 (물리적 배치 및 인터랙션):**
    - 화면 중심 $(c_x, c_y)$를 기준으로 12시 방향부터 시계 방향으로 $\theta = \text{atan2}(x - c_x, c_y - y)$ 공식을 적용해 실시간 좌석 번호(1~N)를 자동 할당합니다.
    - **엔진 상태 `SETUP` (게임 시작 전):** * 두 개의 토큰을 순차적으로 터치(2-Click)하면 캐릭터가 서로 교환(Swap)됩니다. (드래그 앤 드롭 배제)
        - **유연한 재배치 (Re-binding):** 무작위 배분 직후, 이 단계에서 ST는 스크립트 탭에서 지정했던 '위장 대상'의 종속 관계(예: 특정 토큰에 씌워진 주정뱅이 상태)를 다른 토큰으로 유연하게 이동시킬 수 있습니다.
    - **`FIRST_NIGHT` 전환 이후:** 토큰 터치 시 인-게임 행동 팝업 메뉴로 인터랙션이 변경됩니다.
- **스크립트 탭 (풀 구성):** 캐릭터 터치 시 풀에 담기며, 셋업 검증기가 산술 연산을 실시간 구동하여 불일치 시 붉은색 경고문을 출력합니다. 완료 시 마도서 빈 토큰에 역할을 무작위 배분합니다.
- **캐릭터 배분 탭 (오프라인 확인):** 상태 변형이 없는 뷰어 모드입니다. 좌석 번호 1번부터 터치할 때마다 투명도가 80% → 60% → 40%로 낮아지며 은밀한 오프라인 확인을 지원합니다.

**E. 세팅 완료 및 게임 진입 (Initialization & Transition)**

`SETUP` 상태에서 ST가 룰에 맞게 직접 필요한 초기 로직 세팅을 마무리한 뒤 '게임 시작(세팅 완료)'을 누르면, 시스템 엔진 페이즈가 즉각 `FIRST_NIGHT`으로 전환됩니다.

### 3. 능력 사용 UI 스키마 및 입력 처리 규격 (Ability UI Schema & Input Handling)

본 시스템은 개별 캐릭터의 능력 로직과 UI 렌더링 엔진을 완벽히 분리(Decoupling)한다. 프론트엔드 엔진은 캐릭터가 요구하는 스키마(Schema)에 맞춰 화면을 그리고 결과값을 반환할 뿐이며, 데이터의 유효성 검증과 실제 상태 조작은 개별 캐릭터의 JS 파일 내부(`executeAbility`)에서 전적으로 처리된다.

### 3.1. 6대 원자 단위 컴포넌트 (Atomic Components)

모든 캐릭터의 능력 입력 폼은 아래 6가지 기본 스키마의 조합으로만 구성되며, 별도의 하드코딩된 모달 템플릿을 양산하지 않는다.

1. `PLAYER`: 특정 수의 플레이어 지목 (count 속성 포함)
2. `CHARACTER`: 특정 수의 캐릭터 지목 (count 속성 포함)
3. `NUMBER`: 정수 숫자 입력
4. `TEXT`: 자유 텍스트 입력
5. `CHOICE`: ST 지정 단일/다중 선택 (예/아니오, 선/악, 왼쪽/오른쪽 등 커스텀 옵션 배열)
6. `REPEATER`: 복합 입력. 하위 스키마(1~5번) 묶음을 동적으로 N회 반복 추가할 수 있는 컨테이너 (min, max 속성 포함. 예: 곡예사, 가십)

### 3.2. 스토리텔러(ST) 조작 UX 플로우 및 재량권 보장 원칙

시계탑 마피아의 오프라인 감각을 유지하고 ST의 인지적 부하를 줄이기 위해 다음의 UX 플로우를 강제한다.

- **PLAYER 지목 처리 (마도서 본판 연동)**
    - **논블로킹 UI:** 입력 폼은 화면을 가리지 않는 하단/상단 플로팅 패널로 렌더링된다.
    - **시각적 정렬:** 타겟 조건(예: 살아있는 플레이어)에 부합하는 토큰은 밝게 하이라이트 되고, 부합하지 않는 토큰은 어둡게(Dimmed) 처리된다.
    - **직접 터치:** ST는 마도서의 토큰을 직접 터치하여 지목하며, 터치 순서대로 뱃지([1], [2])가 표기된다.
    - **ST 절대 권한:** 어둡게 처리된 '규칙상 예외' 토큰이더라도 ST가 터치하면 시스템은 이를 막지 않고 입력을 허용한다.
- **CHARACTER 지목 처리 (정렬된 리스트 팝업)**
    - 캐릭터 지목 시에는 화면 중앙에 캐릭터 목록 모달이 호출된다.
    - **스마트 정렬:** 현재 플레이 중인 스크립트에 포함되어 있거나, 능력 조건에 부합하는 캐릭터가 목록 최상단에 자동 정렬된다.
    - **ST 절대 권한:** 검색이나 스크롤을 통해 조건에 맞지 않는 외부 캐릭터를 강제 선택하는 조작 역시 시스템이 차단하지 않는다.

### 4. 야간 페이즈 큐(Night Queue) 및 동적 기상 평가 규격

야간 페이즈에 캐릭터가 깨어나는(Wake) 조건은 정적 변수(Boolean)로 통제할 수 없다. 까마귀지기(사망한 당일 밤에만 기상)와 같은 시간적 예외를 처리하기 위해, 시스템 엔진은 개별 예외를 추적하지 않으며 오직 각 캐릭터 클래스 내부에 정의된 '동적 평가 함수'의 반환값에만 의존한다.

### 4.1. 기상 조건 평가 함수 (Predicate Method) 도입

- 모든 캐릭터의 로직은 `BaseLogic`을 상속받으며, 엔진은 야간 큐를 순회할 때 각 캐릭터의 `checkWakeCondition()` 메서드를 호출하여 기상 여부를 묻는다.
- **관심사의 분리 (블랙박스화):** `GameEngine`은 누가 살았고 죽었기 때문에 깨워야 하는지 판단하지 않는다. 그 판단은 전적으로 개별 `CharacterLogic.js`의 책임이다.

### 4.2. BaseLogic 기본 규격 및 오버라이딩 (Overriding)

**기본 규격 (`BaseLogic.js`):**
별도의 예외가 없는 일반적인 캐릭터는 '살아있으면 깨어나고, 죽어있으면 깨어나지 않는' 기본 로직을 따른다.

```jsx
static checkWakeCondition(player, engineState) {
    return player.isAlive;
}
```

**예외 규격 적용 예시 (까마귀지기, 현자 등):**
죽었을 때 능력이 발동하거나 특정 조건에서만 깨어나는 캐릭터는 자신의 클래스에서 해당 메서드를 덮어씌워 엔진의 기본 판단을 무효화한다.

```jsx
static checkWakeCondition(player, engineState) {
    // 예: 살아있으면 깨어나지 않으며, '오늘 밤' 사망자 목록에 자신이 있을 때만 1회 기상
    const diedTonight = engineState.recentDeaths.includes(player.id);
    return !player.isAlive && diedTonight;
}
```

### 4.3. 엔진의 야간 큐 빌드 (Night Queue Build) 로직

야간 순서의 기준은 '현재 플레이어들이 소유한 캐릭터 목록'이 아니라, **'현재 플레이 중인 스크립트(Script) 전체의 정적 밤 순서표(Night Order)'**가 되어야 한다. 이를 통해 야간 페이즈 도중 다른 캐릭터로 변이(예: 마귀할멈의 능력)되었을 때 행동 순서가 누락되는 치명적 오류를 방지한다.

1. `GameEngine`은 밤이 시작될 때, 현재 페이즈(`FIRST_NIGHT` 또는 `NIGHT`)에 해당하는 **스크립트 내 모든 캐릭터의 정렬된 순서 큐(Character ID Queue)**를 불러온다.
2. 엔진은 이 큐를 위에서부터 차례대로 순회하며, 현재 차례의 `Character ID`를 실제로 보유하고 있는 `Player` 인스턴스가 마도서 내에 존재하는지 매번 실시간으로 검색한다.
3. 일치하는 플레이어가 없다면 엔진은 아무 동작 없이 다음 순서로 넘어간다(Skip).
4. 일치하는 플레이어가 존재한다면, 엔진은 순서 실행 직전 해당 플레이어의 `CharacterClass.checkWakeCondition(player, state)`를 호출한다.
5. 반환값이 `true`인 경우에만 해당 순서의 UI 폼을 스토리텔러(ST)에게 호출하고, `false`인 경우 다음 순서로 넘어간다. (만약 동일한 캐릭터를 가진 플레이어가 다수라면, 좌석 번호 순 등 정해진 규칙에 따라 순차 호출한다)

### 5. 주간 행동(Day Action) 진입점 및 처리 규격

낮 페이즈에 발생하는 캐릭터의 능력 사용(예: 처단자, 백치천재 등)은 화면 파편화를 방지하기 위해 마도서(Grimoire) 본판을 통한 단일 진입점으로 통제된다. 능력을 사용하기 위한 별도의 상시 노출 UI나 플로팅 핀(Pin) 패널은 배치하지 않는다.

### 5.1. 단일 진입점 원칙 (Single Entry Point)

- 낮 능력을 지닌 캐릭터의 조작은 밤 페이즈와 완벽하게 동일한 실행 파이프라인(`onRequest` -> UI 렌더링 -> `executeAbility`)을 공유한다.
- 엔진은 주간 행동 처리를 위한 별도의 특수 모달을 만들지 않으며, 기존에 확정된 '6대 원자 단위 컴포넌트' 스키마를 동일하게 호출하여 재활용한다.

### 5.2. UI 상호작용 플로우

1. ST가 낮 페이즈 중 마도서에 배치된 특정 플레이어의 토큰을 터치하여 컨텍스트 메뉴(상태 변경, 처형, 리마인더 부여 등 기본 조작 메뉴)를 호출한다.
2. 해당 토큰의 캐릭터 로직 파일에 `hasDayAbility = true` 속성이 명시되어 있다면, 컨텍스트 메뉴 내에 **[능력 사용]** 버튼이 활성화된다.
3. ST가 해당 버튼을 누르면 엔진은 `onRequest()`를 호출하여 필요한 입력 스키마(예: `PLAYER` 1명 지목)를 화면 하단 패널에 렌더링한다.

### 5.3. 능력 소진(Used) 상태 관리 및 버튼 제어

- 낮 능력(대부분 1회성)의 사용 완료 여부를 판단하고 이후 조작을 차단하는 책임 역시 엔진이 아닌 개별 캐릭터 로직 내부로 격리된다.
- 능력이 정상적으로 실행(`executeAbility`)되면, 로직 내부에서 대상 플레이어(자신)에게 '능력 사용함(used)' 리마인더를 부여한다.
- 엔진은 ST가 토큰을 터치하여 컨텍스트 메뉴를 열 때마다 이 리마인더의 존재 여부를 검사하고, 이미 사용한 상태라면 **[능력 사용]** 버튼을 비활성화(Disabled) 처리하여 중복 사용을 막는다.

### 6. 패시브 및 이벤트성 능력 처리 규격 (Passive & Event-Driven Abilities)

시계탑 마피아의 중독(Poisoned) 및 취함(Drunk) 상태는 ST의 주관적 판단을 요구하므로, 엔진이 특정 이벤트(예: 지목, 사망)를 리스닝하여 능력을 자동 격발(Auto-trigger)시키는 옵저버(Observer) 패턴은 엄격히 배제한다. 모든 능력의 발동은 ST의 수동 개입을 원칙으로 한다.

### 6.1. 순수 패시브 능력 (Passive)

- **대상:** 군인, 시장, 촌장 등 타겟팅의 결과를 수동적으로 변형하거나 상시 적용되는 능력.
- **규격:** `BaseLogic.onRequest()`에서 `UIInputType.NONE`을 반환하거나 아예 입력 폼을 호출하지 않는다.
- **책임:** 엔진은 로직에 개입하지 않으며, ST가 수동 리마인더 마킹 및 타 캐릭터의 능력 처리 과정에서 본인의 판단하에 룰을 반영한다.

### 6.2. 이벤트/반응형 능력 (Event-Driven)

- **대상:** 성결자(지목당함), 돌연변이(광기 위반), 가십(낮 동안의 선언) 등 특정 타이밍에 ST의 판단하에 룰이 개입되어야 하는 능력.
- **규격:** '5. 주간 행동 진입점' 규격을 그대로 재활용하여, ST가 해당 이벤트 발생 시 마도서의 토큰을 터치하여 **[능력 사용]**을 직접 호출하도록 설계한다.
- **책임:** 시스템은 이벤트의 발생 자체를 추적하지 않으며, ST가 물리적 테이블의 흐름을 통제하고 앱의 툴을 수동으로 호출하여 결과를 반영한다.

### 6.3. 구조화된 태그(Structured Tag) 기반 상호작용 및 동적 연산

스토리텔러(ST)의 조작 피로도를 줄이고 히스토리 무결성을 확보하기 위해, 대상에게 영향을 미치는 능력은 시스템이 태그(Tag) 객체를 대조하여 기계적으로 판정한다. 의미론적 충돌을 방지하기 위해 공격과 방어의 속성명(Key)을 엄격히 분리하며, '중독/취함' 상태로 인한 정보 오염을 완벽하게 역추적한다.

**1. 상호작용 태그 객체(Tag Object)의 정의**

- **행동/상태 부여 태그 (Action/Effect):** 능력이 타겟에게 가하는 결과와 수신 제한 조건을 명시한다.
    - 구조: `{ type: "행위타입", validRoles: ["허용대상"] }`
    - 예시 (독술사): `{ type: "POISON", validRoles: ["TOWNSFOLK", "OUTSIDER"] }`
    - 예시 (임프): `{ type: "KILL", validRoles: ["ALL"] }`
- **방어/면역 태그 (Protection/Immunity):** 캐릭터의 패시브나 리마인더로 부여되는 방어 속성이며, 대항하는 위협의 종류를 명시한다.
    - 구조: `{ type: "PROTECT", against: "방어할_위협타입" }`
    - 예시 (수도승): `{ type: "PROTECT", against: "KILL" }`
    - 예시 (선원): `{ type: "PROTECT", against: "ALL" }`

**2. Player 모델의 동적 태그 평가 (`evaluateProtection`)**
대상이 공격이나 상태 이상을 받을 때, `Player.js` 내부의 평가 메서드를 거쳐 방어 태그의 유효성을 검증한다. (요청된 `against` 값을 충족하는지 검사)

- **고유 태그 검사:** 타겟 본인이 중독/취함 상태가 아닐 때만 자신의 패시브 방어 태그를 인정한다.
- **리마인더 태그 검사 (역추적):** 타겟에게 방어 리마인더가 있다면, **해당 리마인더를 부여한 주체(Source Player)**를 역추적하여 그 주체가 중독/취함 상태가 아닌 경우에만 유효한 방어로 인정한다.

**3. 자동 로깅(Auto-Logging) 및 ST 재량권**

- 능력 실행 로직(`executeAbility`)은 평가 메서드 반환값을 바탕으로 기계적 판정을 내리고 결과를 마도서에 자동 기록한다. (예: `[시스템] 임프의 공격이 [패시브 능력(군인)]에 의해 무효화됨.`)
- **ST 절대 권한 보장:** 자동 판정 이후에도 ST는 언제든 마도서 토큰을 수동 터치하여 생사나 상태를 뒤집고 주석(Annotation)을 추가할 수 있다.

### 6.4. 복합 시간축 능력 처리 (Chronological Execution)

알 하디키아와 같이 특정 행위 이후에 시간차를 두고 전체 상태를 재평가해야 하는 복합 능력은, `executeAbility` 내부에서 연산 파이프라인을 단계별로 분리하여 처리한다.

- **1단계 (즉각 적용):** ST가 `REPEATER` 스키마 등을 통해 입력한 배열 데이터(예: 생존/사망 선택)를 즉시 순회하며 개별 생사 및 부활을 1차 처리한다. 이 과정에서 각 대상은 6.3의 동적 태그 평가(`evaluateProtection`)를 거친다.
- **2단계 (사후 조건 평가):** 1단계 연산이 완전히 종료된 시점의 마도서 상태를 기준으로, 사후 격발 조건(예: "현재 전원 생존 중인가?")을 검사한다. 조건 충족 시 추가적인 상태 변화(연쇄 사망 등)를 2차로 집행하며, 이 역시 개별 방어 태그 연산을 독립적으로 통과해야 한다.

### 7. 지목 및 처형 (Nomination & Execution) 시스템의 추후 과제화

투표 및 지목 과정에서 발생하는 다양한 변수(예: 관료의 3표 연산, 골렘의 1회 지목 제약, 성결자의 지목 시 즉사 트리거 등)는 현재 시점에서 시스템 엔진의 태그(Tag)를 통해 섣불리 자동화하지 않는다. 이는 룰의 복잡성을 시스템에 억지로 구겨 넣는 오버엔지니어링(YAGNI)을 방지하기 위함이다.

### 7.1. 스토리텔러(ST) 재량권 우선 원칙

- **UI 통제 금지:** 특정 조건에 따른 투표권 제한(예: 집사)이나 횟수 제한(예: 골렘)을 시스템이 실시간으로 감지하여 지목/투표 UI 버튼을 강제로 비활성화(Disabled) 처리하지 않는다.
- **자동 트리거 금지:** 지목이나 처형 이벤트 발생 시, 시스템이 옵저버(Observer) 패턴으로 이를 가로채어 생사 결과를 자동 격발하지 않는다.
- **통제권의 오프라인 유지:** 위와 같은 제약 및 이벤트 상황은 철저히 오프라인 테이블에서의 ST 육성 통제 및 마도서를 통한 수동 조작 영역으로 남겨둔다.

### 7.2. 백로그(Backlog) 이관

- 기계적인 산술 연산이 요구되는 기능(예: 투표 가치 변형 태그)을 포함한 지목/처형 시스템의 고도화는, 추후 실질적인 개발 및 테스트 단계에서 명확한 한계와 필요성이 입증되었을 때 별도의 명세로 구체화하여 도입한다.

### 8. 징크스(Jinx) 처리 시스템 (추후 과제 / TODO)

특정 캐릭터 쌍이 모였을 때 발생하는 징크스 규칙은 단기적으로는 ST의 수동 판단에 맡기되, 궁극적으로는 캐릭터 능력 로직(`BaseLogic`)과 동일한 수준의 시스템 엔진 개입을 통해 자동화하는 것을 목표로 한다. 단, 구현의 복잡성과 엣지 케이스 폭증을 고려하여 현재 시점에서는 구체적인 연산 명세를 락인(Lock-in)하지 않고 백로그(Backlog)로 이관한다.

### 8.1. 일관성 유지 원칙 (추후 구현 시)

- 징크스는 단순한 텍스트 팝업 경고(Visual Warning)로 끝나는 것이 아니라, 추후 구조화된 태그(Tag) 시스템의 확장이나 `executeAbility` 파이프라인의 간섭(Interception)을 통해 실제 엔진의 연산 결과에 기계적으로 반영되는 설계를 지향한다.

### 9. 전역 상태 저장 및 복구 파이프라인 (Save & Restore)

오프라인 플레이 중 발생하는 기기 방전, 새로고침, 브라우저 강제 종료 등에 대비하여 게임의 전역 상태(State)를 완벽하게 보존하고 복구하기 위한 직렬화 규격이다. 상태 데이터의 단일 진실 공급원(SSOT)은 클라이언트 기기의 내부 저장소가 아닌 로컬 망의 백그라운드 중계 서버(`server.js`)이다.

**1. 즉각적인 파일 I/O 트리거 (Immediate Snapshot)**

- 마도서에서 상태 변동(토큰 이동, 생사 변경, 페이즈 전환, 능력 실행 등)이 발생할 때마다, 중계 서버(`server.js`)는 즉시 최신 전역 상태를 로컬 디렉토리의 `gameState.json` 파일에 덮어쓰기 형태로 물리적 저장한다.
- 상태 데이터의 총 페이로드는 킬로바이트(KB) 수준으로 경량화되어 있으므로 매 행동 시마다 파일 I/O를 발생시켜도 무방하다.

**2. 순수 데이터 직렬화 원칙**

- 상태 객체(`GameEngine.state`)를 직렬화할 때, 클래스 인스턴스 내부의 메서드 및 프로토타입은 구조적으로 보존할 수 없음을 전제한다. 따라서 복구에 필요한 최소한의 순수 상태값과 참조용 ID(`characterId`, `id` 등)만이 스키마에 포함된다.

**3. 복원 및 인스턴스 재조립 (Rehydration & Factory)**

- 중단된 세션을 재개하기 위해 `gameState.json`을 불러올 때, 시스템은 파싱된 단순 객체(POJO)를 엔진에 그대로 주입하지 않는다.
- 시스템은 파싱된 데이터 배열을 순회하며 `new Player(data)` 형태로 주입해 객체 인스턴스를 재조립(Rehydration)한다.
- 이 과정에서 `characterId` 문자열 값을 참조하여 해당 캐릭터의 `BaseLogic` 파생 로직 클래스를 다시 바인딩함으로써, `evaluateProtection` 등 런타임에 필요한 핵심 메서드 구조를 온전히 복원한다.
